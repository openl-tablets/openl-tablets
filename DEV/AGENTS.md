# DEV Module — Core Rules Engine

The foundation of OpenL Tablets. All other modules depend on this. Changes here ripple everywhere.

## Architecture

```
Application Layer (Generated Proxies)  →  Runtime (VM, Context)
  →  Binding (Type Resolution)  →  Parsing (BExGrammar)  →  Source (Excel)
```

**OpenL type system** parallels Java: `IOpenClass` / `IOpenMethod` / `IOpenField` (not Java reflection). Use OpenL types internally; convert at boundaries.

**Binding vs Execution**: compile-time (type checking, method resolution) is strictly separated from runtime. Never mix them.

## Submodules

- **org.openl.rules** — Main engine: type system, parser, binding, bytecode gen, runtime
- **org.openl.commons** — Shared utilities (`org.openl.util`, `org.openl.message`)
- **org.openl.rules.annotations** — Annotation processing for rules
- **org.openl.rules.constrainer** — Constraint solving
- **org.openl.rules.gen** — Code generation for bindings
- **org.openl.rules.project** — Project model, instantiation strategies, resource resolution
- **org.openl.rules.test** — Functional test framework (test table implementations)
- **org.openl.rules.util** — Rule utility functions
- **org.openl.spring** — Spring integration (`PassCoder` for password encoding lives here)

## Critical Systems — Handle with Extreme Care

### Type System (`org.openl.rules/src/org/openl/types/`)

Core interfaces: `IOpenClass`, `IOpenMethod`, `IOpenField`.

Key implementations:
- `AOpenClass` (abstract base, in `types/impl/`)
- `StaticOpenClass` — wraps a Java `Class`
- `ADynamicClass` — base for types built at runtime (in `types/impl/`)
- `ComponentOpenClass` — composite type (in `binding/impl/component/`)
- `JavaOpenClassCache` — caches Java-to-OpenL type mappings

**Rules**: Never change method signatures on `IOpenClass`/`IOpenMethod`/`IOpenField`. Add new capabilities via new interfaces.

### Parser & Grammar (`org.openl.rules/grammar/bexgrammar.jj`)

JavaCC grammar defining OpenL business expression syntax. Generated sources go to `target/generated-sources/`. After grammar changes:

```bash
cd DEV/org.openl.rules && mvn clean compile && mvn test
```

**Gotchas**: Watch for shift/reduce conflicts in JavaCC output. Always test backward compatibility.

### Binding System (`org.openl.rules/src/org/openl/binding/`)

Type resolution, method binding, overload resolution. Performance-sensitive.

- `ANodeBinder` — abstract base for all node binders (in `binding/impl/`)
- `TypeBoundNode` — result of binding with type info (in `binding/impl/`)
- Node binders for XLS tables live in `org.openl.rules.lang.xls.binding/`
- `XlsDefinitions` — registry of table type definitions

### Bytecode Generation (`org.openl.rules/src/org/openl/rules/runtime/`)

Uses ASM (version in root `pom.xml` → `asm.version`). `RulesEngineFactory` is the main entry point for compiling rules. Use `ASMifier` to verify generated bytecode when making changes.

### Project Instantiation (`org.openl.rules.project/src/.../instantiation/`)

- `RulesInstantiationStrategy` — interface for rule loading strategies
- `SimpleProjectEngineFactory` / `SimpleProjectEngineFactory.SimpleProjectEngineFactoryBuilder` — builder pattern for project compilation
- `SimpleMultiModuleInstantiationStrategy` — multi-module loading

## Rule Utility Libraries (`org.openl.rules.util`)

**Registration**: every new class must be added to `org.openl.conf.LibrariesRegistry` via `DEFAULT.addJavalib(YourClass.class)` in its `static` initializer. Forgetting this leaves the methods invisible to rules.

**Boxed-type-only public API**: never expose primitive overloads (`int`, `long`, `float`, `double`, `byte`, `short`). OpenL compiles rule variables to boxed types internally, so primitive variants add no performance benefit and bloat the method-dispatch search space. Keep primitive arithmetic in **private** helpers; public methods accept and return `Byte` / `Short` / `Integer` / `Long` / `Float` / `Double` / `BigInteger` / `BigDecimal`.

## Known Technical Debt

- `JavaOpenClassCache` (`types/java/`) — potential memory leak (needs review)
- `RuleRowHelper` (`rules/binding/`) — string interning may cause memory issues
- `ComponentOpenClass` (`binding/impl/component/`) — calls methods in constructor
- `BinaryNode`, `UnaryNode` (`syntax/impl/`) — public fields should be private
- `TypeBoundNode` (`binding/impl/`) — static method access design issue
- `TypeResolver` (`conf/`) — missing security blocking for system class access
- `CastToWiderType` (`binding/impl/`) — missing generic type support
- `NewArrayNodeBinder` (`binding/impl/`) — incomplete array dimension initialization
